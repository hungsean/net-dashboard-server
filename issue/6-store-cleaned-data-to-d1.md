# 6 - store cleaned data to d1（清洗後資料寫入 D1）

## 背景 / 為什麼

清洗後的資料（#5）要存進 D1，供 GET API（#7）取用。
依 #4 研究的抗變動策略建立 D1 結構與寫入邏輯，讓欄位日後變動時改動成本可控。

## 背景前提
本棒須在 **#4 研究報告經人確認**、且 **#5 清洗層完成**後才實作。

## 範圍

### 要做什麼
- 啟用 `wrangler.jsonc` 的 D1 binding，補上型別宣告。
- 依 #4 建議建立 D1 schema（migration / 建表 SQL），採抗變動策略（如彈性欄位 / JSON column / 混合，以 #4 為準）。
- 新增一個獨立的儲存層模組（例如 `src/data/store.ts`）：輸入 #5 的清洗後結構，寫入 D1。
- 串接流程：upload → (R2 保存) → 清洗(#5) → 寫入 D1。
- 寫入的 idempotency / 更新策略：同一份資料更新時的處理（依 #4 結論，例如 upsert by key），避免髒資料堆積。

### 不做什麼
- 不做 GET 取數 API（那是 #7）。
- 不重做清洗（用 #5）、不改 R2 行為（#3）。
- 不在此棒重新設計資料模型（用 #4 結論）。

## 驗收條件
- [ ] D1 binding 已啟用並有型別宣告，建表 migration / SQL 已就緒。
- [ ] 清洗後資料能正確寫入 D1，可查詢到。
- [ ] 串起 upload → 清洗 → 寫 D1 的完整路徑，端到端可跑通。
- [ ] 同資料更新時依 #4 策略處理（不重複堆積髒資料）。
- [ ] 儲存層為獨立模組，便於後續調整。

中（D1 binding + 建表 + 寫入串接）。
