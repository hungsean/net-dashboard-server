# 5 - clean data transform layer（把原始 json 清洗成符合設計的格式）

## 背景 / 為什麼

上傳的原始 .json 已保存到 R2（#3），但要進 D1 前需先**清洗 / 轉換**成符合設計的結構。
人要求「即使 json 結構日後變動，開發難度也不要過大」→ 本棒依 #4 研究結論，
把清洗/轉換邏輯**獨立成一層**，schema 變動時只改這一層，不擴散到 API 與儲存。

## 背景前提
本棒須在 **#4 研究報告（docs/）出爐並經人確認後**才實作，依其建議的中間結構與分層邊界進行。

## 範圍

### 要做什麼
- 依 #4 的設計，新增一個獨立的清洗/轉換模組（例如 `src/data/clean.ts`）。
- 輸入：原始 payload（來自 upload / R2 的原始 json）。輸出：符合 #4 設計的「清洗後中間結構」。
- 實作清洗規則：欄位對應 / 型別正規化 / 缺漏與異常值處理 / 過濾不需要的欄位（以 #4 報告為準）。
- 邏輯與 API handler、儲存層**解耦**：清洗層不直接碰 R2 / D1，只做純資料轉換（方便測試）。
- 加上單元測試，涵蓋正常與缺漏/異常輸入。

### 不做什麼
- 不寫 D1（那是 #6）；清洗層只回傳清洗後資料，不負責落庫。
- 不改 upload / R2 行為（#2、#3）。
- 不決定資料層設計本身（那是 #4 的結論，本棒只實作）。

## 驗收條件
- [ ] 有一個獨立、可單獨測試的清洗模組（不依賴 R2/D1 binding）。
- [ ] 給定原始 payload → 產出符合 #4 設計的清洗後結構。
- [ ] 缺漏 / 異常 / 多餘欄位的輸入被正確處理（依 #4 規則）。
- [ ] 有單元測試覆蓋正常與異常案例且通過。
- [ ] schema 若變動，改動侷限在此清洗層（設計上可論證）。
